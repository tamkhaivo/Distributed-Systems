## 1. Overview of Updates
Building on the Assignment 2 Client-Server application, the system was updated to introduce a shared **server.log** file. A `simulation.sh` script was created to spawn 1 `Server` process and 5 concurrent `Client` processes. The clients randomly generate `READ` and `WRITE` requests to interact with the file over TCP sockets. 
- A `WRITE` appends timestamped data to the file.
- A `READ` returns the full current state of the file to the client.

## 2. Approach to Communication and File Access
Because multiple clients access the same `Server` process concurrently via `ClientHandler` threads, accessing the file directly would cause **race conditions** (e.g., interrupted writes or interleaved data).

To solve this, I leveraged java's `ReentrantReadWriteLock`. 
- When a `WRITE` command is received, the thread requests a **Write Lock** (`rwLock.writeLock().lock()`). This guarantees exclusive access, blocking any other reads or writes while appending the text.
- When a `READ` command is received, the thread requests a **Read Lock** (`rwLock.readLock().lock()`). Multiple clients can hold read locks simultaneously, maximizing concurrency, as long as no write lock is currently held. 

## 3. Issues with Increasing Processes
If we scale this simulation from 5 clients to 5,000 clients, the following issues will occur:
1. **Thread Starvation/Overhead**: The server creates a `new Thread()` for every client. Thousands of threads will exhaust the server's CPU and memory via context switching. 
2. **File I/O Bottleneck**: Disk write speeds (IOPS) are significantly slower than network speeds. All 5,000 clients funneling into a single `server.log` file will cause massive queuing at the `WriteLock`, generating high latency.
3. **Single Point of Failure**: If the `Server` process crashes due to Out-Of-Memory, or if the `server.log` disk is corrupted, the entire system halts.

## 4. Application of Coordination Algorithms
Based on **Module 10: Coordination**, our current implementation uses a **Centralized Mutual Exclusion Algorithm**. The server's `ReentrantReadWriteLock` acts as the single coordinator managing the queue for the lock.

As discussed in the module, centralized algorithms suffer from single points of failure and performance bottlenecks. If we wanted to deploy this system at scale:
*   We should use **ZooKeeper** (Section 5.3.6). It handles distributed locking perfectly via ephemeral sequential znodes.
*   Alternatively, to prevent the single-file bottleneck and ensure fault tolerance, we could use a cluster of servers utilizing a consensus algorithm like **Raft** (Section 5.4.4) to replicate the log securely across multiple machines rather than a single `server.log`.
